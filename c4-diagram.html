📊 Typischer Datenfluss: „Ausgabe hinzufügen"

1
AddExpenseForm validiert Eingabe → ruft onAdd(expense) auf
2
useStore.addExpense() – optimistisch: setExpenses([neu, ...prev]) → sofortige UI-Aktualisierung
3
supabaseClient – async: supabase.from('expenses').insert({...}) → HTTP POST an REST API
4
PostgreSQL speichert den Eintrag → Realtime-Event wird ausgelöst
5
Zweiter Client (Lisa) empfängt WebSocket-Push → fetchExpenses() → UI-Update auf Lisas Handy
6
Bei Fehler: Rollback (setExpenses(prev.filter(...))) + opError-Banner

Bonus – Datenbankschema (Supabase)

Vier Tabellen mit Row Level Security (anon-Zugriff, keine Authentifizierung).

💸
Tabelle
expenses
id uuid PK
description text
amount numeric(10,2)
category_id text
paid_by text (person1|2)
split_ratio numeric(5,4)
date date
receipt_image text (base64)
notes text
created_at timestamptz
⚙️
Tabelle (Singleton)
settings
id smallint PK (=1)
person1_name text
person2_name text

Immer genau 1 Zeile.
CHECK (id = 1)
📦
Tabelle
kassensturz
id uuid PK
created_at timestamptz

Trennmarke zwischen aktiven und archivierten Ausgaben.
Kein FK – Zeitstempel-basierte Logik.
🛒
Tabelle
shopping_items
id uuid PK
text text
checked boolean
created_at timestamptz

Unabhängig von Expenses. Vollständiger CRUD + Realtime.
🔐 Sicherheitsmodell: Alle Tabellen haben Row Level Security aktiviert. Policies erlauben dem anon-Role (öffentlicher Supabase-Key) SELECT, INSERT, UPDATE und DELETE – ohne Benutzerauthentifizierung. Das Zugriffskontrollmodell setzt auf Geheimhaltung der App-URL, nicht auf echte Authentifizierung.
